# PayFlow API

A small REST backend for a UPI-style payment app. It lets you register users, give
them a wallet balance, look them up, and record money transfers between them. There is
no UI — everything is driven over HTTP.

Built with Spring Boot 3.5, Spring Web, Spring Data JPA and an in-memory H2 database.

## Running it

You need JDK 21. From the project root:

```bash
./mvnw spring-boot:run
```

The app starts on `http://localhost:8080`. The H2 web console is at
`http://localhost:8080/h2-console` (JDBC URL `jdbc:h2:mem:payflow`, user `sa`, empty
password).

Because the database is in-memory, the tables are recreated on every start and the data
is gone once you stop the app.

## How the project is organised

The code is split into four packages, each with one job:

- **entity** — `User` and `Transaction`. Plain Java classes annotated with `@Entity`;
  JPA turns them into database tables.
- **repository** — `UserRepository` and `TransactionRepository`. Interfaces extending
  `JpaRepository`. Spring Data writes the actual SQL for us.
- **service** — `UserService` and `TransactionService`. The business logic lives here.
  The controllers talk to services, not to repositories directly.
- **controller** — `UserController` and `TransactionController`. They map HTTP requests
  to methods and return JSON.

A request flows in one direction: `controller → service → repository → database`.

## Where Spring Boot's three features show up here

**Embedded server.** There is no Tomcat to install or configure. The `spring-boot-starter-web`
dependency ships Tomcat inside the build, so `./mvnw spring-boot:run` boots a server on
port 8080 by itself. The startup log line `Tomcat started on port 8080` is this in action.

**Auto-configuration.** I never wrote a single line of database wiring. Because H2 and
Spring Data JPA are on the classpath, Spring Boot auto-configures a `DataSource`, a
Hibernate `EntityManager` and the H2 console for me. Just adding the dependencies was
enough — that is `@SpringBootApplication` (which includes `@EnableAutoConfiguration`)
doing the work.

**Production-ready defaults.** Sensible defaults are already in place without any setup:
a connection pool (HikariCP, visible in the startup log), JSON serialisation via Jackson,
and a baked-in port and error handling. I only overrode the few properties I actually
cared about in `application.properties`.

## Endpoints

| Method | Endpoint              | What it does                                 |
|--------|-----------------------|----------------------------------------------|
| POST   | `/users`              | Register a user (JSON body), returns the saved user |
| GET    | `/users`              | List all users                               |
| GET    | `/users/{id}`         | Get one user by id                           |
| GET    | `/users/upi/{upiId}`  | Get one user by their UPI id                 |
| POST   | `/transactions`       | Record a money transfer (JSON body)          |

### Sample curl calls

```bash
# register two users
curl -X POST http://localhost:8080/users -H "Content-Type: application/json" \
  -d '{"name":"Priya Sharma","upiId":"priya@okaxis","balance":5000.0,"phoneNumber":"9876543210"}'

curl -X POST http://localhost:8080/users -H "Content-Type: application/json" \
  -d '{"name":"Rahul Verma","upiId":"rahul@oksbi","balance":2000.0,"phoneNumber":"9123456780"}'

# look them up
curl http://localhost:8080/users
curl http://localhost:8080/users/1
curl http://localhost:8080/users/upi/rahul@oksbi

# send money
curl -X POST http://localhost:8080/transactions -H "Content-Type: application/json" \
  -d '{"senderUpiId":"priya@okaxis","receiverUpiId":"rahul@oksbi","amount":250.0,"note":"dinner split"}'
```

Output of these calls and the H2 console screenshots are in [`docs/WRITEUP.md`](docs/WRITEUP.md).

## The CREATE TABLE statements (Task 2)

With `spring.jpa.show-sql=true`, Hibernate prints the DDL it generates on first startup:

```sql
create table users (
    user_id bigint generated by default as identity,
    balance float(53),
    name varchar(255),
    phone_number varchar(255),
    upi_id varchar(255),
    primary key (user_id)
);

create table transactions (
    transaction_id bigint generated by default as identity,
    amount float(53),
    note varchar(255),
    receiver_upi_id varchar(255),
    sender_upi_id varchar(255),
    primary key (transaction_id)
);
```

Notice the camelCase Java fields (`upiId`, `phoneNumber`) became snake_case columns
(`upi_id`, `phone_number`) — that is JPA's default naming strategy, no annotation needed.

> Note on table names: `USER` and `TRANSACTION` are reserved words in H2, so the entities
> are mapped to `users` and `transactions` with `@Table`. The H2 queries are therefore
> `SELECT * FROM USERS` and `SELECT * FROM TRANSACTIONS`.

## findByUpiId — the derived query (Task 3)

`UserRepository` declares:

```java
User findByUpiId(String upiId);
```

I never wrote its body. When the app runs, Spring Data generates this SQL:

```sql
select u1_0.user_id, u1_0.balance, u1_0.name, u1_0.phone_number, u1_0.upi_id
from users u1_0
where u1_0.upi_id = ?
```

**(a) How JPA derives it from the method name.** Spring Data parses the method name into
parts. `findBy` means a SELECT, and `UpiId` is matched against the entity's `upiId` field
(which maps to the `upi_id` column). So `findByUpiId` becomes "select the user whose
`upi_id` equals the argument". The method name *is* the query.

**(b) What the `?` means.** It is a bind parameter (a placeholder). The query is sent to
the database as a prepared statement with a `?` where the value goes, and the actual UPI
id is bound separately at execution time. This is what prevents SQL injection and lets
the database reuse the same query plan for different values.

## Three ways to write a custom query (Task 6)

`UserRepository` shows two of the three approaches:

```java
User findByUpiId(String upiId);                                  // derived method name

@Query("SELECT u FROM User u WHERE u.balance > :amount")          // @Query with JPQL
List<User> findRicherThan(@Param("amount") Double amount);
```

1. **Derived method names** — you just name the method (`findByUpiId`) and Spring Data
   writes the SQL. Best for simple lookups; no SQL to maintain, reads like English. It
   gets unwieldy for anything complex.
2. **`@Query` with JPQL** — you write the query yourself, but against *entities and fields*
   (`User`, `u.balance`), not tables and columns. Good when the logic is too involved for a
   method name. It is still database-independent because Hibernate translates the JPQL to
   the right SQL dialect.
3. **Native SQL** (`@Query(..., nativeQuery = true)`) — raw SQL against real table and
   column names.

**Why native is least preferred:** it ties your code to one specific database's SQL. If
you ever switch databases the query may break, and you lose the type-safety and entity
mapping that JPQL/derived queries give you. You only reach for it when you genuinely need
a database-specific feature that JPQL cannot express.
